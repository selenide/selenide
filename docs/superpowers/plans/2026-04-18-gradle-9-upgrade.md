# Gradle 9.x Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade Gradle wrapper from 8.14.4 to the latest 9.x, restore broken custom browser test tasks, and clean up Gradle-9-deprecated build-script conventions.

**Architecture:** Three small, sequential edits:
1. Bump wrapper distribution URL.
2. Wire `testClassesDirs`/`classpath` into the common `Test` configuration block in `gradle/tests.gradle` (fixes the empty-discovery problem for all 11 custom browser test tasks).
3. Remove redundant `sourceCompatibility` declarations in `gradle/compilation.gradle` now that toolchain alone suffices. Validate after each edit.

**Tech Stack:** Gradle 9.x, Gradle Wrapper, Java 17 toolchain, SpotBugs, SonarQube, vanniktech Maven Publish, JUnit 5, TestNG.

**Spec:** `docs/superpowers/specs/2026-04-18-gradle-9-upgrade-design.md`

## Task 5: Sweep remaining Gradle 9 deprecations

**Files:** Determined by the warning scan in Step 1. Likely candidates in order of probability: `build.gradle`, `gradle/javadoc.gradle`, `gradle/coverage.gradle`, `gradle/code-analysis.gradle`, and per-module `build.gradle` files under `modules/*/` and `statics/`.

- [ ] **Step 1: Capture every remaining deprecation**

Run: `./gradlew clean check --warning-mode all --console=plain 2>&1 | tee /tmp/gradle9-deprecations.log | grep -iE '(deprecat|is scheduled to be removed|has been deprecated)' | sort -u`
Expected: Zero or more unique deprecation messages. Read each one.

- [ ] **Step 2: For each deprecation, identify the source line**

For each deprecation from Step 1, re-run with `-s` (stacktrace) to get the script location:

Run: `./gradlew clean check --warning-mode all -s --console=plain 2>&1 | grep -B2 -A10 '<deprecation-message-substring>'`
Expected: Output identifies the build-script file and line number that triggered the deprecation. If the deprecation originates from a plugin (not from our own scripts), note the plugin version — addressed in Task 6.

- [ ] **Step 3: Fix each in-repo deprecation**

For every deprecation originating from a `.gradle` file under the repo, apply the recommended replacement. Typical Gradle 9 fixes:

- `archivesBaseName = 'x'` → `base { archivesName = 'x' }`
- `project.hasProperty('foo')` used as a task-config-time read → move to a `Provider` pattern if surfaced.
- `JavaExec.main` → `JavaExec.mainClass`.
- `convention` methods → extensions or services.

If an unexpected deprecation appears that isn't covered here, look up the Gradle 9 upgrade guide section matching the message text and apply its recommendation. Do not silence warnings — fix root causes.

- [ ] **Step 4: Re-run the scan and iterate until the in-repo deprecation list is empty**

Run: `./gradlew clean check --warning-mode all --console=plain 2>&1 | grep -iE '(deprecat|is scheduled to be removed|has been deprecated)' | sort -u`
Expected: Any remaining messages originate from third-party plugins (handled in Task 6), not from files under `gradle/` or the repo's `build.gradle` files.

- [ ] **Step 5: Commit**

```bash
git add -A
git commit -m "clear Gradle 9 deprecation warnings from in-repo build scripts"
```

If there were no in-repo deprecations to fix, skip the commit — the previous commits already left the tree clean.
