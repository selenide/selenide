# Gradle 9.x Upgrade ‚Äî Design

**Date:** 2026-04-18
**Status:** Approved
**Scope:** Upgrade Gradle wrapper from 8.14.4 to the latest 9.x, restore custom browser test tasks (`chrome_headless`, `firefox_headless`, etc.), and clean up deprecated build-script conventions surfaced by Gradle 9.

## Problem

After bumping the Gradle wrapper to 9.x, the custom test tasks defined in `gradle/tests.gradle` (e.g. `chrome_headless`, `firefox_headless`, `chrome`, `firefox`, `edge`, `edge_headless`, `edge_headless_smoke`, `chrome_headless_smoke`, `chrome_remote`, `firefox_remote`, `safari`, `ie`) do not execute any tests. The default `test` task (for unit tests) continues to work.

## Root cause (hypothesis)

In Gradle 9, `Test`-typed tasks created via `tasks.register("name", Test)` no longer inherit `testClassesDirs` and `classpath` from the `test` source set automatically. The custom tasks therefore run with empty test-discovery roots, so no tests are found or executed.

Validation during implementation will confirm via `./gradlew chrome_headless --dry-run` that, before the fix, no integration tests are listed.

## Fix strategy

### 1. Gradle wrapper bump

- **File:** `gradle/wrapper/gradle-wrapper.properties`
- **Change:** Bump `distributionUrl` to the latest stable Gradle 9.x release at implementation time. Use `./gradlew wrapper --gradle-version <x>` run twice (first run updates `gradle-wrapper.properties`, second run updates `gradle-wrapper.jar` and `gradlew`/`gradlew.bat` if needed).

### 2. Explicit source-set wiring in Test tasks

- **File:** `gradle/tests.gradle`
- **Change:** In the existing common `tasks.withType(Test).configureEach { ‚Ä¶ }` block (lines 114‚Äì136), add:

  ```groovy
  testClassesDirs = sourceSets.test.output.classesDirs
  classpath       = sourceSets.test.runtimeClasspath
  ```

- **Why this location:** Every Test task ‚Äî the default `test` plus all 11 custom ones ‚Äî gets it without per-task duplication. Setting these for the default `test` task is redundant but harmless.

### 3. Clean up deprecated conventions

- **File:** `gradle/compilation.gradle`
- **Changes:**
  - Remove the first `java { sourceCompatibility = '17' }` block (lines 13‚Äì15). The toolchain block below already covers language level.
  - Remove the per-`JavaCompile` `sourceCompatibility = 17` line (line 9). Redundant with toolchain.
  - If Gradle 9 emits deprecation warnings for `options.debug` or `options.debugOptions.debugLevel`, address inline.

### 4. Plugin sanity check (no code changes expected)

Plugins that already look Gradle 9 compatible ‚Äî verify at build time and bump only if they emit deprecations:

- `com.github.spotbugs 6.5.0`
- `org.sonarqube 7.2.3.7755`
- `com.vanniktech:gradle-maven-publish-plugin 0.36.0`

## Validation plan

Run in this order, fixing problems before moving on:

1. `./gradlew check` ‚Äî unit tests green.
2. `./gradlew chrome_headless --dry-run` ‚Äî confirm the task resolves integration test classes (output lists integration tests, not empty).
3. `./gradlew chrome_headless` ‚Äî full integration run.
4. `./gradlew javadocForSite` ‚Äî per `CLAUDE.md` gate.
5. `./gradlew allTests` ‚Äî final belt-and-braces (clean + check + firefox_headless + chrome_headless).
6. Re-run with `--warning-mode all` once and address any remaining deprecations inline.

## Out of scope

- Java toolchain version bump (stays at 17).
- Selenium / JUnit / other library version bumps.
- Refactoring the 11 custom test tasks into a helper (that was option B from brainstorming).
- Migration to `jvm-test-suite` plugin (would be a larger, independent change).

## Risks

- The `testClassesDirs`/`classpath` assignment assumes the `java` plugin is applied on every subproject running Test tasks. It is (see `build.gradle:24`), so this is safe.
- A plugin may emit new Gradle 9 deprecations not anticipated here. Addressed inline during validation step 6.
- If `firefox_headless` requires Firefox on the CI/local machine and it is missing, the validation step will fail for a non-Gradle reason; treat such failures separately.

## Files touched

- `gradle/wrapper/gradle-wrapper.properties` (+ possibly `gradle-wrapper.jar`, `gradlew`, `gradlew.bat` regenerated by the wrapper task)
- `gradle/tests.gradle`
- `gradle/compilation.gradle`
- Possibly `build.gradle` (only if a plugin version needs bumping)
